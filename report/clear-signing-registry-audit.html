What is clear signing?

Without clear signing, hardware wallet users see raw hexadecimal data when approving transactions. A uint256 named _value carries no hint that it should display as "100 USDT" instead of 100000000. Users are effectively signing blind.

ERC-7730 is a JSON descriptor format that tells wallets how to present transaction data in human-readable form. Each descriptor defines an intent (the headline action like "Swap" or "Repay loan"), labels (field names like "Amount to Send"), and formats (how to render values with proper decimals, token symbols, dates).

The Ledger registry is where dApp teams submit these descriptors via pull request. They are imported into Ledger device firmware and used whenever a user signs a transaction for a registered contract.

The key fields that control what users see:

intent is the headline. "Swap", "Supply", "Repay loan". label on each field is the name next to the value. "Amount to Send", "Beneficiary". format determines rendering: tokenAmount shows "100 USDT", addressName resolves ENS, date converts timestamps.

Protocols where the clear signing labels would actively confuse or mislead users on the hardware wallet screen.

Positive patterns

What works well

Not everything is broken. These protocols demonstrate that consistent labeling is achievable at scale.

ERC-2612 Permit (87 files)

Perfectly consistent across every token: Owner Spender Amount Deadline Nonce. The same five labels, identically capitalized, across 87 files. This is the best-standardized UX pattern in the entire registry and proves that shared templates work.

Morpho / Kiln Vaults (100+ files)

Share common definitions via includes. Labels are uniform across all vault variants: Assets Shares Owner. The includes mechanism in ERC-7730 enables this reuse pattern.

Safe (25 files)

Thorough coverage across Safe 1.3.0, 1.4.1, 1.5.0, SafeL2, and SafeMigration. Sensible, actionable labels: New Owner New threshold Transaction Module. Descriptive without being verbose.
